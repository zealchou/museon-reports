01

市場定位分析

這套系統定位精準——它不是 CRM，不是業績報表，而是一套專為「行為紀律追蹤」設計的工具。 保險業的核心管理挑戰，在於「到底每個業務有沒有做到基本功」，而非只看結果數字。

填補的市場空缺

🎯 行為追蹤 ≠ 業績追蹤
市場上 95% 的系統追蹤的是「結果」（保費、FYC、成交數）。本系統追蹤的是「過程行為」——每日三訪、早會出席、問卷完成。

藍海定位 行為 → 結果的因果鏈，是主管真正需要的洞察。
🟢 紅綠燈介入機制
一般系統資料是靜態的，主管需要自己看、自己判斷。紅綠燈自動評分（綠/黃/紅）讓主管只需關注「需要介入」的人員，大幅降低管理認知負荷。

效率槓桿 10 人團隊節省主管每週 3–4 小時追蹤時間。
🔒 資料主權優勢
保險業監理嚴格，業務行為資料屬高敏感。架設在客戶自己的伺服器，資料不經第三方 SaaS，完全符合金管會合規要求與內控需求。

合規加分 SaaS 方案普遍難以通過法遵審查。

目標客群定義

主要購買者：保險公司區部主管 / 處經理，管理旗下業務 10–50 人，過去靠 LINE 群組 + Excel 追蹤，每週花大量時間催打卡、整理資料、在早會唸名單。

核心痛點：「我知道誰沒做基本功，但我要花太多時間去確認和記錄。」

次要使用者：業務員本人（每日回報）、公司稽核部門（合規紀錄）、HR（人員留任分析）。

系統定位矩陣

這系統 是什麼
  • ✓ 業務每日行為打卡工具
  • ✓ 主管紅綠燈異常介入系統
  • ✓ 照片佐證 + 稽核紀錄工具
  • ✓ FYC 目標進度追蹤工具
  • ✓ 法遵自查記錄工具
這系統 不是什麼
  • ✗ 客戶 CRM（不儲存客戶個資）
  • ✗ 業績試算 / 佣金試算系統
  • ✗ 保單管理系統
  • ✗ 核保 / 理賠系統
  • ✗ 帳務 / 收費系統

06

談判重點建議

從客戶立場設計談判策略——哪些功能是 v1 必做的高價值核心，哪些可留後階段，以及合約中必須保護的權益。

v1 必做（高價值，不可妥協）

MUST 紅綠燈評分引擎
這是整個系統的核心差異點。沒有紅綠燈，系統就只是「數位版 LINE 打卡」，無法讓主管快速判斷異常。

評分邏輯需可彈性設定（各指標加權比例），不能是固定公式。
MUST 主管儀表板（今日總覽）
主管使用最高頻的功能。今日有幾人達標、幾人黃燈、幾人紅燈，必須在一個畫面看完。

若儀表板不夠直觀，使用率將大幅下降，影響系統價值感知。
MUST 任務設定靈活性
不同保險公司、不同主管對「基本功」定義不同。任務必須可設定、可加減項目，否則系統無法跨客戶複用或因應各家特殊規定。
MUST 照片佐證 + 審核流程
保險業主管最怕業務「填假資料」。照片佐證是系統公信力的來源。

審核工作流（業務提交 → 主管審核/退回）是管理閉環的關鍵，必須在 v1 完成。

可留到 v2（降低 v1 阻力）

  • LINE 推播提醒 — v1 可先用 Email 或系統內通知替代，v2 再接 LINE Notify / LINE Bot。
  • 自動化報表排程發送 — v1 主管手動匯出即可，v2 加排程自動寄送 PDF 報表。
  • 行動 APP 優化 — v1 做響應式網頁版（手機瀏覽器使用），v2 再做 PWA 或原生 APP。

可留到 v3（進階功能，有數據才有意義）

  • AI 行為趨勢分析 — 分析哪些行為組合最能預測 FYC 達標，需要半年以上的數據累積才有統計意義。
  • 人員留任風險預測 — 根據行為分數下滑趨勢預測離職風險，屬高階 HR analytics。
  • 多團隊 / 跨區比較分析 — 集團型保險公司才需要，一般 10–50 人團隊 v1 已足夠。

合約保護點（必談）

📜 原始碼交付條款
  • 原始碼完整交付給客戶，存入指定 Git repository，確保不被單一廠商綁定。
  • 交付時須含部署文件（README、環境需求、安裝步驟），讓客戶 IT 或第三方能接手維護。
💾 資料備份 & 復原 SLA
  • 明訂備份頻率（至少每日一次），備份保留天數建議 90 天。
  • 指定復原 RTO：若主機故障，服務需在 24–48 小時內恢復。
🚪 退出機制
  • 終止合約時廠商需提供資料匯出工具（CSV / JSON 格式），確保資料可攜性。
  • 維護合約每年可重新議約，不強制綁定多年，確保費率合理。
🔒 資安 & 合規要求
  • 明訂不儲存客戶個人資料（個資法保護條款），只追蹤業務行為。
  • 傳輸加密 HTTPS 強制，密碼儲存需 bcrypt / Argon2，不得明文。
