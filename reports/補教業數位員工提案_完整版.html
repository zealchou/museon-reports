PAIN POINTS

補教業的三大挑戰

💰

人力成本持續攀升

一位行政人員每月平均花費 60% 以上的時間在回覆重複查詢、整理點名表、處理繳費通知。這些工作不需要人的判斷力，卻佔用了最有價值的人力資源。

估算：4 間分校 × 2 位行政 × 60% 時間浪費 = 相當於每年 80–120 萬的人力成本花在可自動化的工作上。

🎯

服務品質難以標準化

家長的問題在下班後、週末發送，等到隔天才有人回覆——這段空窗期，競爭對手可能已經成交。招生窗口、課程說明的話術也因人而異，品質不穩定。

🧠

40 年經驗無法有效傳承

老師的招生技巧、課程設計邏輯、處理特殊學生的方式，都存在個人腦袋裡。一旦老師離職，這些寶貴知識就跟著帶走，新人需要重新摸索數年。

DATA SECURITY

資料安全設計原則

資料主權（Data Sovereignty）

🔒 核心原則

學生資料、家長個資、財務紀錄都儲存在您自己的環境中——本地伺服器或私有雲。AI 在您的圍牆內工作，不把您的資料傳出去分析。

On-Premise Storage Private Cloud

身份與存取控管（IAM）

🛡️ RBAC

每位使用者有專屬帳號與角色權限。行政人員看不到財務報表，老師看不到其他班級的家長聯絡資料——Role-Based Access Control 確保敏感資訊不越界。

Role-Based Access Control

傳輸加密（TLS 1.3）

所有資料傳輸全程 TLS 1.3 加密，確保家長與系統之間的對話不被截聽或竄改。

TLS 1.3

儲存加密（AES-256）

資料庫靜態加密，即便儲存媒介遺失，資料也無法被第三方讀取。

AES-256

操作稽核日誌（Audit Log）

所有系統操作留下完整紀錄，查看誰在什麼時間做了什麼，符合個資法規範。

Compliance Log

SECURITY TECHNOLOGY

Agent 採用的六大安全機制

🧱
沙箱隔離（Sandbox Isolation）
每個 Agent 運行在獨立執行環境中，招生 Agent 的操作範圍被嚴格限制，無法存取教務或財務資料。即使單一 Agent 出問題，也不會影響其他功能。
🔑
零信任存取（Zero-Trust Architecture）
任何請求都需要驗證身份，不假設內部網路是安全的。每次操作都驗證：你是誰、你有沒有權限、這個請求合不合理。
📋
個資去識別化（Data Anonymization）
AI 在進行統計分析時，使用去識別化資料而非原始個資——分析結果有意義，但即使被截取，也看不出是哪位學生或家長的資料。
🔍
即時威脅偵測（Anomaly Detection）
系統持續監控異常行為——短時間大量查詢、非正常時段操作、試圖存取越權資料——自動觸發警報並封鎖可疑請求。
♻️
資料保留政策（Retention Policy）
依照貴校需求設定資料保留週期，過期資料自動安全刪除，確保不保留不必要的個資，符合個資法第五條規定。
💾
容災備份（Disaster Recovery）
定時快照備份，Recovery Point Objective (RPO) 設計為 < 1 小時，確保系統即使遇到硬體故障，也能在極短時間內恢復正常運作。
