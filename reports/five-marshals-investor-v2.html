MUSEON 五虎將

五個自主維運引擎，讓 AI 顧問系統在無人值守下持續運行、自我監測、自動修復——將「人力運維成本」從固定支出變為零。

解決什麼問題？

AI 系統的運維悖論

一個 AI 顧問系統要 24/7 服務客戶，但傳統做法需要一支 DevOps 團隊來監控、修復、優化—— 人力成本吃掉了 AI 帶來的效率紅利。

五虎將的設計命題是：讓 AI 系統自己照顧自己。不是取代人類判斷，而是把「重複性巡檢、已知故障修復、品質抽查」這些標準化工作交給自動化引擎，讓人類只處理真正需要決策的問題。

五虎將是誰？

五個角色各司其職，形成「觀察 → 偵測 → 審計 → 修復 → 巡檢」的閉環。每個引擎都是獨立的 Python 模組，零耦合、可獨立升級。

🔭

MuseWorker — 觀察員

Observer & Recorder
24/7 背景常駐的護理師。不修改任何東西，只觀察、記錄、計算。為其他四虎提供最新的系統快照。
  • 即時記錄 Skill 鍛造、Workflow 執行、拓撲變動
  • 計算全系統扇入表（216+ 模組依賴關係）
  • 產出 snapshot.json 供其他引擎消費
  • 變動日誌追溯——誰在什麼時候改了什麼
執行週期：每 3 小時 零 LLM 調用
🚨

MuseOff — 巡邏糾察隊

7-Layer Probe Engine
7 層探測（L0-L6）的即時防禦系統。發現問題立即止損，留下結構化診斷卡。CPU 效能上限 30%，不拖慢主服務。
  • L0: Gateway 存活探針
  • L1: 服務就緒檢查（Telegram / LLM / DB）
  • L2-L4: 模組健康、記憶一致性、連線完整性
  • L5-L6: Baseline 偏移偵測、趨勢異常警報
執行週期：每 5 分鐘 零 LLM 調用
🔍

MuseQA — 品質審計官

Quality Auditor
每 15 分鐘掃描所有對話紀錄，確保 AI 回覆符合設計規範。CPU 預篩 90%（正則匹配），LLM 抽檢 10%。
  • 術語洩漏偵測——內部架構用詞不可出現在回覆中
  • 人格偏移檢查——回覆風格是否偏離顧問定位
  • 跨群組污染掃描——A 客戶資訊不可洩漏到 B 客戶
  • 回覆品質閘門——過短、過長、空洞的異常偵測
執行週期：每 15 分鐘 LLM 成本 ~$0.04/天
🔧

MuseDoc — 外科醫生

DSE Surgeon
每夜 4am 啟動的自動修復引擎。讀取所有 finding，用 Runbook 閉環修復。五道安全防線確保「修好不修壞」。
  • Runbook 匹配——沒有對應方案則標記 needs_human
  • 扇入檢查——高影響模組自動拒絕修改
  • 術前快照（git stash）+ 術後驗證
  • 每夜上限 3 修 / 回滾 ≥ 2 次立即停工
執行週期：每日 04:00 每夜最多修 3 個
🛡️

Guardian — 巡檢衛士

Infrastructure Guard
每 30 分鐘進行基礎設施巡檢。檢查配置完整性、外部服務連線、常見故障模式，並嘗試自動恢復。
  • 環境配置（.env）完整性驗證
  • Telegram / LLM 服務連通性探測
  • 常見故障模式偵測與自動修復
  • 異常結果即時推送 Telegram 通知決策者
執行週期：每 30 分鐘 零 LLM 調用

運作流程

五虎將形成一條完整的「發現 → 診斷 → 修復 → 驗證」管線。每個環節獨立運作，透過結構化的 Finding 檔案傳遞資訊。

閉環修復管線

1
MuseWorker 建立基線 每 3 小時掃描全系統 216+ 模組，產出扇入表、連線狀態、共享狀態快照。這是所有偵測的基準線。
2
MuseOff 即時探測 7 層探針每 5 分鐘巡邏一次，對照 MuseWorker 的基線偵測偏移。發現異常立即產出 Finding 診斷卡。
3
MuseQA 品質抽檢 每 15 分鐘掃描對話紀錄，用 27 個正則模式預篩 + LLM 抽檢，捕捉術語洩漏、人格偏移、跨群污染。
4
MuseDoc 夜間手術 凌晨 4:00 讀取所有 Finding，匹配 Runbook，在 5 道安全防線保護下自動修復。修不了的標記 needs_human。
5
Guardian 持續巡檢 每 30 分鐘驗證修復結果是否生效，基礎設施是否健康。異常時推送 Telegram 通知決策者。

達成的效果

五虎將讓一個 5 人的 AI 顧問團隊不需要額外的 DevOps 人力，就能維持 24/7 的系統運行。

🔒 運維自動化

  • 零人力值守 — 巡檢、偵測、修復全自動化
  • 結構化診斷 — 每個問題都有 Finding 卡，含根因分析與影響範圍
  • 安全修復 — 5 道防線（Runbook / 扇入 / 快照 / 驗證 / 上限）
  • 決策分流 — 能自動修的自動修，需要人判斷的才通知

🛡️ 品質保障

  • 資訊隔離 — 跨客戶資料洩漏零容忍偵測
  • 術語防漏 — 27 個正則模式攔截內部架構用詞
  • 人格一致性 — AI 回覆風格持續對標設計規範
  • 異常回覆警報 — 過短/過長/空洞內容即時標記

💰 成本結構

  • 95% 純 CPU — 五虎將絕大部分工作不需要 LLM
  • LLM 成本 ~$0.04/天 — 僅 MuseQA 抽檢使用最小模型
  • 取代 DevOps 人力 — 省下 $15K-50K/月的運維團隊成本
  • 可預測支出 — 自動化排程 = 固定的日/月計算成本

📊 可觀測性

  • 完整日誌鏈 — 從 Finding → Triage → Surgery 全程追溯
  • 系統快照 — 每 3 小時一次全量掃描存檔

給投資人的話

五虎將不是一個「完美無缺的系統」——它是一個能自我監測、自我改善的系統。 我們的護城河不在於「每個零件有多強」，而在於「失效時誰來救」這個閉環設計。

上面列出的改善項目，本身就是五虎將運行過程中自己發現的——MuseOff 偵測到 Telegram 斷線，MuseDoc 發現自己的 Runbook 覆蓋不夠——這正是自主維運系統的價值：它不只運行，還會告訴你哪裡需要變得更好。

隨著客戶數量增加，五虎將的成本不會線性成長。一台引擎照顧 1 個客戶和 100 個客戶的計算成本幾乎相同——這是 AI 顧問業務可規模化的技術底層。

技術架構參考

🧱 設計原則

  • CPU 優先 — 能用本地計算就不呼叫 LLM
  • 零耦合 — 每個引擎可獨立部署、升級、停用
  • 結構化輸出 — Finding / Prescription / Surgery 全 JSON
  • 防禦性設計 — 每個 job 獨立 try/except，單點失敗不擴散

⚙️ 技術棧

  • 排程引擎 — APScheduler + asyncio 非同步並行
  • 靜態分析 — 自研 FanInScanner（AST 解析 216+ 模組）
  • 修復引擎 — Runbook 閉環（pre-check → action → post-check → rollback）
  • 品質引擎 — 27 正則模式 + Haiku LLM 抽檢
